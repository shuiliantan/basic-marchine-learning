现在，机器学习在很多问题上，都已经超越了人类的表现，比如

  • 用户点击某个广告的概率
  • 产品推荐
  • 预估从A地开车到B地的时间

假设人类的表现约等于贝叶斯误差

  • 偏差bias：当算法的表现和人类表现存在巨大差距时，意味着欠拟合，推荐消除偏差，可以尝试更大的神经网络或者训练更长时间，提升模型在训练集上的性能；

  • 方差：当算法的表现接近人类的表现时，就希望重点减少方差

2020-05-13

为什么人们会把机器学习系统和人类表现比较

  • 效果：随着深度学习发展，机器学习效果迅速提高，使得在很多新领域的算法和人相比也更有竞争力
  • 效率：在某些领域用机器学习系统解决问题的比人工更高效

![image-20200513095440631](/Users/shuiliantan/Library/Application Support/typora-user-images/image-20200513095440631.png)

对于很多机器学习任务，在算法接近人类表现之前，都进展的较快。当算法超过人类表现时，进展和准确性的提升就减慢了。再继续训练算法时，效果逼近从来不会超过1个理论值（贝叶斯最优误差），这是因为

  • 人们的表现大多数情况下接近 贝叶斯最优误差
  • 当你超过人类表现时，挺难有工具去帮助提高算法表现

当 算法表现没有人好时

  • 可以去标记数据
  • 分析为什么人标记是对的，但是算法弄错了
  • 分析偏差和方差

2020-05-12

结论：

  • 当评估指标，无法正确的对算法的优劣排序时，就需要更换评估指标了。
  • 验证集或者测试集上表现很好，但不能在真正关心的应用场景上也表现的很好时，就需要去更改dev/test set或者metric

对团队的建议：设立一个明确的目标，有效率的朝着改善性能的方向进行迭代

  • step1: place the target，先定义评估指标
  • Step 2: aim at the target，再考虑如何在这个指标上做好；比如定义损失函数，优化求解等

举例1：项目进行到一半，发现目标（评估指标）弄错了。例如构建一个猫的分类器，把分类为猫的图片展示给爱猫的用户。刚开始设置分类准确率为评估指标，算法A 和算法B在验证集上的分类错误率分别为 3%和5%，从评估指标和验证集出发，应该选择算法A。但是后来发现，算法A会把色情图片错分为猫推荐给用户，算法B不会，那么从公司角度出发，算法A是绝对不允许的，应该选择算法B。这时：评估指标已经不能对算法的优劣进行排序了，所以我们应该修改评估指标。比如当错分类为色情图片时，权重由原来的1变为10，100等。

举例2：猫分类器，算法A和算法B的分类错误率分别为3%和5%，训练集和验证集都是取景高清的猫的图片。但是在线上使用分类器时，算法B的效果更好。因为用户都是上传模糊的低质量图片。用户关心的是系统能不能识别出他们的图片。这个时候，就应该去修改开发集测试集，让开发集测试集的数据能够更好反应实际数据的情况。

在现代的深度学习中，我们拥有更大量的数据，使用远小于20%或者30%的数据作为开发集或者测试集是非常合理的。因为深度学习算法极需要数据 ，训练集会占有更大的比例。

测试集：作用就是帮我们评估最终系统的性能，因此，测试的大小只要能保证对系统整体性能评估的高置信度即可。

有时候没有测试集也是可以的，不过不建议这么做，最好在上线前使用测试集得到对系统性能的无偏估计

总的来说：传统的认为传统的70/30法则已经不再适用了；现在的趋势是 使用更多数据作为训练集 使用较少的数据作为开发集和测试。

dev set的作用是：针对训练集训练很多不同的模型之后，利用dev set去评估，然后挑选1个好的模型

建议dev 和test set 来自同一分布，因为这样你们就瞄准了目标，可以让团队更有效率。不然会浪费团队的时间。

不然，可能团队花了好几个月的时间，努力的靠近dev set，在最后测试的时候，效果并不好。

比如：一个团队花费数月时间，优化dev set，数据为中产地区贷款审批信息。任务为：根据这些信息，预测他是否能拿到贷款，以及为什么。做了几个月之后，团队忽然决定在低收入地区测试这个模型，模型效果很差。只能把之前做过的一些工作重新做一遍。为什么这样呢？这就像过了三个月，老板问另一个目标怎么样了，对于团队来说是很恼火的。

So, that dev set is also called the development set, or sometimes called the hold out cross validation set. And, workflow in machine learning is that you try a lot of ideas, train up different models on the training set,

and then use the dev set to evaluate the different ideas and pick one.

结论：当想要把所有的指标合成1个数值类型的指标并不容易时，可以选择1个指标加以优化，让它尽可能的表现好，剩下的N-1个指标，让他们达到某个阈值就好了，一旦达到阈值，就不再费心优化这N-1个指标了。

  • optimizing metric （优化指标）：想要这个指标尽可能的好，例如accurary；
  • satisficing metric（满足指标）: 只需要达到阈值就可以，并不关心在阈值范围内有多好，例如running time 。

举例：

![image-20200508130656732](/Users/shuiliantan/Library/Application Support/typora-user-images/image-20200508130656732.png)

It’s not always easy to combine all the things you care about into a single row number evaluation metric.

In those cases I’ve found it sometimes useful to set up satisficing as well as optimizing matrix.

So more generally, if you have N matrix that you care about it’s sometimes reasonable to pick one of them to be optimizing. So you want to do as well as is possible on that one. And then N minus 1 to be satisficing,

meaning that so long as they reach some threshold such as running times faster than 100 milliseconds,

but so long as they reach some threshold, you don’t care how much better it is in that threshold, but they have to reach that threshold.
